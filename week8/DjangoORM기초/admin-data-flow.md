# Admin과 데이터 흐름 정리

## 사용자 CRUD와 Admin CRUD의 역할 차이

사용자 CRUD는 `/lions/` 화면을 통해 일반 사용자가 아기사자 목록을 조회, 등록, 수정, 삭제하는 흐름을 담당한다. 화면 흐름은 템플릿 기반으로 제공하며, POST 요청 이후에는 `redirect()`로 응답해 중복 제출을 방지한다.

Admin CRUD는 `/admin/` 화면에서 운영자가 데이터를 관리하는 흐름을 담당한다. 운영자는 같은 `Lion` 모델 데이터를 목록 컬럼, 검색, 트랙 필터, 최신순 정렬, 페이지네이션을 활용해 빠르게 확인하고 관리할 수 있다.

## 운영 관점에서 Admin이 필요한 이유

서비스 운영자는 사용자 화면보다 더 넓은 관리 권한과 빠른 탐색 기능이 필요하다. Django Admin은 별도의 사용자 화면을 만들지 않아도 모델 데이터를 추가, 수정, 삭제할 수 있고, 검색과 필터를 통해 운영 중 필요한 데이터를 빠르게 찾을 수 있게 한다.

## SQL과 ORM의 차이

SQL은 개발자가 테이블, 컬럼, 조건을 직접 문장으로 작성하여 DB에 요청하는 방식이다. ORM은 `Lion` 같은 Model 클래스를 통해 Python 객체를 다루면 Django가 필요한 SQL로 변환하여 DB에 요청하는 방식이다. 이 프로젝트에서는 `cursor.execute()` 같은 직접 SQL 호출을 사용하지 않고, `Lion.objects.create()`, `all()`, `get()`, `filter()`, `order_by()`, `save()`, `delete()`로 데이터를 처리한다.

## Lazy Loading이 필요한 이유

QuerySet은 조회 조건을 담는 객체이며 생성되는 즉시 DB를 조회하지 않는다. 템플릿에서 순회하거나 `list()`, `count()`처럼 결과가 필요한 시점에 쿼리가 실행된다. 이 지연 실행 덕분에 `filter()`와 `order_by()` 같은 조건을 이어 붙인 뒤 필요한 최종 쿼리만 실행할 수 있어 불필요한 DB 접근을 줄일 수 있다.

## 6주차, 7주차, 8주차의 차이

| 항목 | 6주차 메모리 기반 | 7주차 Model + DB 기반 | 8주차 ORM 이해 |
| --- | --- | --- | --- |
| 저장 위치 | 코드 내 전역 리스트 | MySQL DB | MySQL DB |
| 재시작 후 유지 | 유지되지 않음 | 유지됨 | 유지됨 |
| 데이터 구조 | `dict` | `Lion` 모델 인스턴스 | `Lion` 모델 인스턴스와 QuerySet |
| 조회 방법 | 리스트 순회 | 기본 ORM 조회 | `filter()`, `get()`, `order_by()`, `Q`, `count()` |
| 핵심 관점 | CRUD 흐름 경험 | Model과 Admin 도입 | ORM 동작 원리와 데이터 영속성 이해 |

가장 큰 차이는 데이터를 바라보는 수준이다. 6주차는 메모리 안의 값을 직접 다루고, 7주차는 Model과 DB로 영속성을 확보한다. 8주차는 같은 Model 기반 구조에서 QuerySet이 언제 DB에 접근하고 어떤 조건으로 데이터를 가져오는지 이해하는 데 초점을 둔다.
